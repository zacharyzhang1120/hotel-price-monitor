# 酒店竞对价格监控系统 — 模块化开发计划

> 版本：v1.1 | 日期：2026-06-30 | 目标：优先快速做出可测试 MVP，再逐步替换真实抓取能力

---

## 当前实现状态

- P0 MVP 闭环已完成：种子酒店、Mock 抓取、价格日历、当日对比、微信文本日报、手动刷新、Chrome 插件均已跑通。
- P0 联调已完成：`scripts/smoke_test.py` 覆盖健康检查、酒店映射、抓取批次、日历、日报、备份。
- P1 已具备接入骨架：携程真实抓取器、平台开关、真实 URL 探测脚本与 API、映射查看/更新脚本、平台映射配置 API、插件内平台 URL 配置、单链接探测、单酒店刷新已就绪。
- P2 已具备基础能力：抓取历史、抓取配置接口、手动备份、备份列表、定时任务开关和状态接口、时间配置、插件内运行状态/备份面板、可选企业微信机器人推送已就绪。
- 多门店组能力已进入测试版：每个我方酒店可维护最多 5 家竞对；切换我方酒店后，巡检、配置、抓取、日报、就绪检查都按当前门店组执行。
- 携程 URL 自动匹配已可用：支持单酒店自动匹配，也支持“当前组一键匹配”，已有 URL 默认跳过，不覆盖人工配置。
- 携程真实抓取已进入测试版基线：阿里云 `SCRAPER_MODE=real`、`SCRAPE_CONCURRENCY=1`、`scope=today` 时，6 家酒店今日价格最近后台批次 Batch 50 达到 6/6 成功，耗时约 4 分 41 秒。去哪儿、同程暂未进入真实抓取验收。

## 一、总体策略

### 1.1 MVP 优先原则

首个可测试 MVP 不依赖真实 OTA 抓取成功，而是先用种子酒店 + 模拟价格数据跑通完整闭环：

```
酒店配置 → 抓取批次 → 价格入库 → 价格 API → 日报生成 → 浏览器插件展示 → 手动刷新
```

这样可以在最短时间内验证：
- 数据模型是否支持真实业务口径
- 前端热力图、当日对比、运营判断和证据链是否好用
- 日报价格口径是否符合收益经理决策习惯
- 后续真实抓取器接入时，是否只需要替换数据来源

### 1.2 模块化与回滚原则

- 每个模块独立开发、独立验收、独立提交。
- P0 阶段所有平台抓取器先接入 MockScraper，真实 OTA 抓取按平台逐个替换。
- 每个平台抓取器单独文件、单独开关、单独日志，某个平台失败不影响其他平台。
- 数据库结构先一次性定稳：`hotels`、`hotel_platform_mappings`、`scrape_runs`、`price_records`。
- API 响应结构先定稳，前端只依赖 API，不直接关心抓取来源。

## 二、优先级总览

| 优先级 | 目标 | 结果 |
|---|---|---|
| P0 | 快速 MVP 闭环 | 可手动刷新、可看热力图/对比表、可生成微信文本日报 |
| P1 | 接入真实抓取 | 携程先稳定，去哪儿、同程逐个替换 Mock 数据源 |
| P2 | 稳定性与自动化 | 证据链、定时任务、备份、错误告警、抓取重试、二期企业微信推送 |

## 三、P0：MVP 闭环（建议 3-5 天）

### P0.1 后端基础与数据模型（0.5-1 天）

**目标**：先把底座搭稳，避免后面频繁返工。

交付内容：
- FastAPI 应用入口、CORS、配置文件。
- SQLAlchemy async 数据库连接。
- ORM 模型：
  - `hotels`
  - `hotel_platform_mappings`
  - `scrape_runs`
  - `price_records`
- 初始化数据库表。
- 基础健康检查接口：`GET /health`。

验收标准：
- 后端可以启动。
- SQLite 数据库自动创建。
- 4 张核心表存在，索引和唯一约束生效。

回滚边界：
- 只影响数据库与后端基础结构，不涉及前端和真实抓取。

---

### P0.2 种子数据与 Mock 抓取器（0.5 天）

**目标**：不等待 OTA 抓取，先制造稳定测试数据。

交付内容：
- `scripts/seed_hotels.py`
- `scripts/seed_data.json`
- `MockScraper`
- 手动生成一批今日 + 未来 7 天价格数据。
- 每次 Mock 抓取都生成独立 `scrape_run` 批次。

验收标准：
- 能插入 1 家我方 + 5 家竞对。
- 每家酒店有 3 个平台映射。
- 手动执行 Mock 抓取后，生成 `6 × 3 × 8 = 144` 条价格记录。
- 连续执行多次后，可以看到同一入住日的历史价格变化。

回滚边界：
- MockScraper 独立文件，不影响未来真实平台抓取器。

---

### P0.3 价格 API（0.5-1 天）

**目标**：先稳定前端和日报依赖的数据接口。

交付接口：
- `GET /api/v1/hotels`
- `GET /api/v1/prices/calendar`
- `POST /api/v1/scrape/trigger`
- `GET /api/v1/scrape/status/{task_id}`
- `GET /api/v1/scrape/runs/{batch_id}/tasks/{task_result_id}/evidence`

关键规则：
- `calendar` 默认读取最新成功批次。
- `calendar` 支持 `batch_id` 指定批次。
- `calendar` 返回 `cheapest_room` 和 `cheapest_price`。
- `trend` 支持不传 `platform`，返回同一酒店多平台历史线。
- 手动抓取先调用 MockScraper。

验收标准：
- API 能返回 8 天日历数据。
- API 能返回某酒店某入住日的多平台趋势数据。
- 手动刷新接口不阻塞请求，状态可轮询。

回滚边界：
- API 层与抓取层通过服务接口隔离，后续替换真实抓取器不改变前端。

---

### P0.4 ReportService 与微信文本日报（0.5 天）

**目标**：先验证业务口径和推送文本可读性。

交付接口：
- `POST /api/v1/reports/generate?format=json|wechat_text|wechat_markdown`
- `GET /api/v1/reports/latest`

关键规则：
- 我方基准价 `baseline_price` = 我方各平台最低起价。
- 竞对总览 `vs_mine` = 竞对最低价 - 我方最低起价。
- 平台差价 `vs_mine_same_platform` = 竞对某平台价格 - 我方同平台价格。
- `wechat_text` 使用缩进列表，不使用 Unicode 表格线。

验收标准：
- 能生成 JSON 报告。
- 能生成企业微信手机端友好的纯文本日报。
- 示例中竞对 A 同程 ¥628 vs 我方最低 ¥668，`vs_mine = -40`。

回滚边界：
- 报告服务只依赖价格 API/数据库，不影响抓取器。

---

### P0.5 浏览器插件 MVP（1-1.5 天）

**目标**：让用户尽快看到可用界面。

交付内容：
- WXT + Vue 3 插件项目。
- Popup 主界面。
- 组件：
  - `CalendarHeatmap`
  - `ComparisonTable`
  - `HotelSelector`
  - `RefreshButton`
  - `StatusBadge`
  - `MappingConfigPanel`
  - `OperationsPanel`
- API composables：
  - `useApi`
  - `usePrices`
  - `useScrape`

MVP 界面只做必要功能：
- 打开插件自动加载最新价格日历。
- 支持切换全部/单个竞对。
- 点击日历单元格聚焦该天价格明细。
- 点击“立即刷新”触发 Mock 抓取并刷新数据。
- 当日对比表支持复制文本。
- 支持在插件中维护平台 URL、默认房型，并进行 Mock/真实单链接探测。
- 支持在插件中查看定时状态、最近备份，并手动创建数据库备份。

验收标准：
- 插件能在本地浏览器加载。
- 无真实 OTA 抓取时，也能完整展示数据。
- 手动刷新后，热力图和当日对比数据变化。

回滚边界：
- 前端只依赖固定 API 合约；真实抓取失败不会影响前端开发。

---

### P0.6 MVP 联调与修正（0.5 天）

**目标**：把前后端闭环跑顺，形成第一版可测试产品。

验收清单：
- 启动后端。
- 初始化酒店数据。
- 触发一次 Mock 抓取。
- 打开插件看到 8 天价格日历。
- 点击单元格看到对应日期价格明细。
- 生成微信文本日报。
- 手动刷新后数据更新。

MVP 完成定义：
- 收益经理不用看代码，仅通过插件就能完成一次价格巡检。
- 开发者可以通过报告 API 拿到可推送文本。
- 后续接入真实平台抓取器时，不需要重写数据模型和前端。

## 四、P1：真实平台抓取（建议 5-8 天）

### P1.1 抓取器基础框架（0.5 天）

交付内容：
- `BaseScraper`
- `PricePoint`
- 抓取器注册表
- 平台开关配置：
  - `SCRAPER_MODE=mock|real|mixed`
  - `ENABLED_PLATFORMS=ctrip,qunar,tongcheng`
- 单平台失败日志与错误记录。

验收标准：
- Mock 与真实抓取器可以通过配置切换。
- 某个平台关闭后，不影响其他平台抓取。

---

### P1.2 携程抓取器（1.5-2 天）

开发顺序：
1. 先抓取单酒店单日期。
2. 再扩展到单酒店 8 天。
3. 最后纳入 ScraperManager 批量抓取。

验收标准：
- 能返回 `check_in_date`、`cheapest_room`、`cheapest_price`。
- 单日期失败不影响其他日期。
- 携程失败时可以回退到 Mock 或跳过携程。

回滚策略：
- 只回滚 `ctrip.py` 和平台配置，不影响去哪儿、同程、API、前端。

---

### P1.3 去哪儿抓取器（1-1.5 天）

开发顺序：
1. URL 日期参数验证。
2. 单酒店单日期抓取。
3. 单酒店 8 天抓取。
4. 接入批量抓取。

验收标准：
- 与 BaseScraper 返回结构一致。
- 平台失败不影响其他平台。

回滚策略：
- 只回滚 `qunar.py` 和平台配置。

---

### P1.4 同程抓取器（1-1.5 天）

开发顺序：
1. 优先验证移动端页面或移动端接口。
2. 单酒店单日期抓取。
3. 单酒店 8 天抓取。
4. 接入批量抓取。

验收标准：
- 与 BaseScraper 返回结构一致。
- 同程抓取失败时批次仍可完成，状态标记为 partial_success。

回滚策略：
- 只回滚 `tongcheng.py` 和平台配置。

---

### P1.5 真实抓取联调（1-2 天）

目标：
- 将真实平台数据接入 P0 前端和报告。
- 验证价格口径、房型名称、失败处理。

验收标准：
- 至少 1 家我方 + 1 家竞对可以跑通真实抓取。
- 至少 1 个平台可以稳定跑通 8 天。
- 失败平台不会阻塞整批数据入库。

---

## 五、P2：稳定性与自动化（建议 3-5 天）

### P2.1 定时调度（0.5 天）

交付内容：
- APScheduler 启动/关闭。
- 当前线上每日 08:00、11:00、14:00、17:00、20:00、23:00 自动抓取今日价。
- 定时任务只抓当前有效门店组，避免历史孤立酒店拖慢后台任务。
- 调度开关 `SCHEDULER_ENABLED`。

验收标准：
- 手动和定时抓取都生成 `scrape_run`。
- 同一时间不会重复启动多个批次。
- 配置页能看到定时目标家数、最近定时耗时和健康状态。

---

### P2.2 备份与日志（0.5-1 天）

交付内容：
- 每日 SQLite 备份到 `data/backups/`。
- 抓取错误日志。
- 批次错误摘要。

验收标准：
- 可追溯某次抓取失败的平台、酒店和日期。
- 备份文件可恢复。

---

### P2.3 抓取稳定性增强（1-2 天）

交付内容：
- 单日期重试。
- 平台级超时。
- 随机延迟。
- 并发限制。
- partial_success 状态。

验收标准：
- OTA 某一页卡住不会拖死整个批次。
- 抓取统计能区分成功、失败、跳过。

---

### P2.4 企业微信推送（二期）（1 天）

交付内容：
- 企业微信推送通道配置。
- 抓取完成后推送 `wechat_text`。
- 推送失败不影响抓取批次成功状态。

验收标准：
- 企业微信手机端文本不乱版。
- 推送内容与报告 API 一致。

## 八、风险排序

| 风险 | 影响 | 应对 |
|---|---|---|
| OTA 反爬导致真实抓取不稳定 | 高 | P0 先用 Mock 完成产品闭环，P1 再逐平台替换 |
| 数据模型返工 | 高 | P0.1 先落实酒店实体、平台映射、抓取批次 |
| 微信日报手机端错位 | 中 | 使用缩进文本，不使用 Unicode 表格 |
| 前端等待真实数据导致进度卡住 | 中 | 前端只依赖 Mock/API 合约 |
| 单个平台失败拖垮整批任务 | 中 | 平台独立开关、独立错误记录、partial_success |

## 九、MVP 验收口径

MVP 必须满足：
- 可以配置 1 家我方 + 5 家竞对。
- 可以生成今日 + 未来 7 天价格。
- 可以展示价格日历热力图。
- 可以展示当日价格对比表。
- 可以点击查看历史趋势。
- 可以手动刷新生成新批次。
- 可以生成企业微信友好的日报文本。
- 可以在插件里配置平台 URL 和默认房型。
- 可以在插件里查看定时状态并手动备份。

MVP 暂不要求：
- 真实 OTA 抓取全部稳定。
- 企业微信实际发送成功。
- 复杂权限认证。
- 多用户管理。
- 云端部署。
